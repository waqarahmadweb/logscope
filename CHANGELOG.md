## [0.9.0] - 2026-04-28

Closes Phase 11 of the [roadmap](ROADMAP.md): polish, accessibility, dark mode, and i18n. The Logscope admin page picks up its first dedicated stylesheet (variable-driven for light / OS-dark / `admin-color-midnight`), loading skeletons replace the bare spinner on initial fetch, REST failures and save successes surface as `@wordpress/components` Snackbars from a new `toasts` slice, four global keyboard shortcuts (`/`, `g`, `t`, `?`) land alongside WAI-ARIA-compliant tablist arrow-key navigation, and `languages/logscope.pot` now covers all 87 PHP and JS strings. With Phase 11 done the next milestone is the v1.0.0-rc.1 cut: `readme.txt`, the wp.org banner / icon / screenshot set, and the release workflow that strips dev assets from the published zip.

### Added

-   Phase 11.1–11.5: polish, accessibility, dark mode, and i18n. The Logscope admin page picks up its first dedicated stylesheet (`assets/src/style.scss`, imported from the React entry); previously every `logscope-*` class name was a semantic hook with no rules attached and the page inherited whatever wp-admin shipped, which left rows visually undifferentiated and gave dark-mode users on `admin-color-midnight` an effectively unstyled UI. The new sheet is variable-driven so component rules reference `--logscope-*` tokens exclusively and every theme variant — light, OS dark via `prefers-color-scheme`, and the WP `admin-color-midnight` admin scheme — is one variable-block remap rather than a parallel rule set, keeping the cascade as the single source of truth. A `prefers-reduced-motion` block disables the skeleton shimmer for users who opt out.
-   `ListSkeleton` / `FormSkeleton` components (Phase 11.1) — replace the bare `<Spinner />` previously rendered on the initial fetch with shimmering placeholder rows that hint at the eventual shape of the list (or form). Spinners gave no sense of layout, so the page looked broken between the request kick-off and the first byte; skeletons keep the user oriented. Both wrappers render `role="status"` + `aria-live="polite"` so screen readers announce the loading state, while the inner skeleton bars are `aria-hidden` to avoid the AT layer narrating a stack of empty divs.
-   Toast / Snackbar surface (Phase 11.1) — new `toasts` slice on the `logscope/core` store + a `ToastHost` component mounted at the App root that renders the queue as `@wordpress/components` Snackbars. The `pushToast` action is fired from every REST failure path in the store (`fetchLogs`, `saveSettings`, `testLogPath`) so a transport blip surfaces as an explicit cue rather than just a silent `error` field on the slice; settings save also fires a `success` toast so the admin gets confirmation without having to read the page for state diff. Each toast carries a self-pruning `expiresAt` timestamp the host turns into a single-effect timer chain, and `dismissToast` is wired to the Snackbar's `onRemove` so click-to-dismiss races cleanly with the auto-prune.
-   Empty-state copy is now filter-aware (Phase 11.1). `EmptyState` takes a `filtersActive` flag and the LogViewer derives it from the live filter slice; instead of a generic "No log entries to show." an admin staring at the screen after a typo in the regex sees "No entries match the current filters. Try widening the date range or clearing the regex." — actionable rather than a dead end. The error variant keeps `role="alert"` for immediate AT announcement; the empty variant uses `role="status"` with a polite live region.
-   Keyboard shortcuts (Phase 11.2). New `useKeyboardShortcuts` hook bound at App level: `/` focuses the regex search input (also selects the existing value so a power user can replace-by-typing), `g` toggles list ↔ grouped, `t` toggles tail mode on/off, and `?` opens a help modal listing the bindings. Shortcuts are gated on `event.ctrlKey || metaKey || altKey` (so `Ctrl+T` still opens a browser tab) and on the focused element being a non-editable target (so typing `g` into the regex field is a literal `g`, not a view toggle). The hook is decoupled from the components that handle each shortcut: App emits a `logscope:shortcut` `CustomEvent` and `LogViewer` / `FilterBar` subscribe to the events that target their DOM. This keeps the global key listener (which has to live above the tab switch) free of knowledge about specific refs and lets each handler colocate with the state setters or focus targets it manipulates. Pressing `g` or `t` from the Settings tab auto-switches to the Logs tab first, so the keystroke can't be silently dropped on the wrong screen. A "Shortcuts (?)" button in the tab bar opens the same modal for users who don't read changelog entries.
-   `HelpModal` component (Phase 11.2). Built on `@wordpress/components` Modal, which gives focus trapping, escape-to-close, and `aria-labelledby` for free. The shortcut list is a `<dl>` of `<kbd>` elements; the trailing note ("Shortcuts are ignored while typing in an input.") explains the gating so users don't think `?` is broken when nothing happens from inside the regex field.
-   Accessibility pass (Phase 11.3). Tab buttons now carry roving `tabIndex` (active = 0, inactive = -1) so a keyboard user can `Tab` through the page without landing on every inactive tab. The regex search input picks up an explicit `aria-label` ("Search log messages (regex)") because the visible "Search (regex)" label disappears under the field at narrow widths; the aria copy is the durable contract. The skeleton wrappers and the empty-states wire `role="status"` / `role="alert"` + `aria-live` regions appropriately so AT users get the same loading / empty / error cues a sighted user does. A global `:focus-visible` ring lands inside `.logscope-app` so the whole admin page picks up a consistent 2px outline (instead of relying on whatever wp-admin's reset leaves in place); the ring uses the `--logscope-focus-ring` variable and remaps in dark mode along with the rest of the palette. The `Shortcuts` toggle button carries an `aria-label` distinct from its visible "Shortcuts (?)" caption so screen readers don't read out the literal punctuation.
-   Dark-mode parity (Phase 11.4). `@media (prefers-color-scheme: dark)` flips the entire palette to a dark variant tuned against `#1d2327` (WordPress's canonical dark-admin background) — text, borders, severity-pill tones, skeleton shimmer, focus ring, and the Snackbar host all rebase off the same `--logscope-*` variables. A `body.admin-color-midnight` selector additionally overrides the OS preference for users who explicitly picked WordPress's only first-party dark admin scheme, since "I picked midnight in my profile" is a stronger signal than `prefers-color-scheme`. The light/dark severity-pill tones are tuned for WCAG AA contrast on each background so a `Fatal error` pill stays readable in both modes.
-   i18n pass (Phase 11.5). Audited every user-facing string across `assets/src/**` and `src/**`; all toast messages, skeleton aria-labels, shortcut descriptions, and the help-modal copy are wrapped in `__()` / `_n()` / `sprintf()` with translator notes where placeholders carry meaning. New `languages/logscope.pot` (87 messages, both PHP and JS strings — neither `wp-pot-cli` nor `wp-scripts` extracts both languages in one pass, so the build uses WP-CLI's `i18n make-pot`). Generated by a new `pnpm i18n` script that downloads `wp-cli.phar` to a gitignored `tools/` directory on first run rather than pulling `wp-cli/wp-cli-bundle` (~150 MB of Composer dev deps); PHP must be on PATH but the plugin already requires PHP ≥ 8.0 for development. The script lives in `bin/make-pot.mjs` (export-ignored, so it does not ship in the wp.org zip).

### Changed

-   Phase 11 review follow-ups. **Tabs arrow-key navigation** — `App` now binds `onKeyDown` on the `role="tablist"` element so Left / Right cycle between tabs and Home / End jump to the first / last; previously a keyboard user could only `Tab` into the active tab and `Tab` back out, which left a real gap against the Phase 11.3 "keyboard nav for all interactive elements" promise (axe-core wouldn't have caught it — the structure was correct, the behaviour wasn't). Each tab carries a `data-logscope-tab` attribute the handler queries on the next frame to move focus to the newly-active tab, since the roving `tabIndex` flips on the very next render. **Shortcut-event constants extracted** — the `SHORTCUT_EVENT` name and the three event-detail strings (`focus-search`, `toggle-grouped`, `toggle-tail`) now live in `assets/src/shortcuts.js` rather than being re-exported from `App.jsx` and inlined as string literals across `LogViewer` and `FilterBar`. The previous shape worked through ESM hoisting (App imported the leaves and the leaves imported `SHORTCUT_EVENT` back from App), but the bidirectional graph was brittle and the inline literals were typo-prone. **HelpModal switch removed** — the previous shape stored English description strings in the array and ran them through a `describe()` switch that mapped each one back to a `__()` call. Adding a fifth shortcut without updating the switch would have silently fallen through to the untranslated string. The array now inlines `__()` directly at each entry; make-pot extracts them at the call site. **ToastHost wrapper aria-live removed** — `@wordpress/components` Snackbar already announces via its own aria-live region; the wrapper's `aria-live="polite"` would have caused some screen readers to double-announce. The wrapper now styles only.
-   `AssetLoader::enqueue()` now resolves the stylesheet basename. `@wordpress/scripts` (since v23) emits the CSS chunk for an entry `index` as `style-index.css` rather than `index.css`; the previous code looked for the legacy filename and would have silently skipped the new stylesheet. The loader now tries `style-index.css` first and falls back to `index.css` so a hand-built bundle following the older convention still enqueues. `AssetLoaderTest::test_enqueue_includes_style_when_css_present` updated to assert the new filename, and a new `test_enqueue_falls_back_to_legacy_index_css` covers the fallback branch.

## [0.8.0] - 2026-04-28

Closes Phase 8 of the [roadmap](ROADMAP.md): the Settings tab is now a real form. Admins can edit `log_path` and `tail_interval`, save through the existing `POST /settings` route, and probe a candidate log path through a new side-effect-free `POST /settings/test-path` endpoint that surfaces `PathGuard`'s verdict inline — including a clean fall-through to a parent-directory writability check so a fresh install pointing at a not-yet-created custom path gets a green answer instead of a misleading "does not exist" rejection. With the editor in place the Phase 7.4 tail loop now has a UI for changing its cadence, and the custom log path the rest of the codebase has been validating since Phase 3 finally has a way to be configured from wp-admin. Phase 11 (polish, a11y, i18n) is the next milestone before the v1.0.0-rc.1 cut.

### Added

-   `SettingsPanel` component (Phase 8.1) — the Settings tab in `App.jsx` now renders a real form built on `@wordpress/components` (`TextControl` × 2, `Button`, `Notice`, `Spinner`) backed by a new `settings` slice in the `logscope/core` store. The slice keeps `draft` and `values` separate so the admin can edit freely and Reset without mutating server-of-record state; Save POSTs the whole draft, the server returns the new authoritative shape, and the reducer copies it into both slots. The slice also carries `loadError` / `saveError` / `fieldErrors` / `testResult` / `testError` / `isTesting`, and the dispatcher gains `*fetchSettings` / `*saveSettings` / `*testLogPath` generator thunks consistent with the existing `*fetchLogs` shape. Per-field rejections from the existing `unknown_setting` 400 surface as inline notices under the offending input (mapping `error.data.unknown[]` into a `fieldErrors` map in the catch arm) rather than just a top-level banner. The dirty check trims `log_path` before comparing so a trailing space doesn't toggle Save into a dirty state for a no-op edit, matching what the server-side sanitiser would have stored anyway. Field-error wording is held in the panel layer (`translateFieldError(code)`) so the store records codes (`'unknown_setting'`) rather than untranslated strings — every user-facing string still goes through `__()`.
-   Custom log path UI + side-effect-free path probe (Phase 8.2). New `POST /wp-json/logscope/v1/settings/test-path` REST route on the existing `SettingsController`, gated by the same `permission_callback` as the rest of the surface (`logscope_manage` cap, 401 / 403 from the abstract base). The endpoint always returns 200 with a structured verdict body — a malformed candidate is a normal answer to a probe, not a request error — shaped `{ ok, resolved, exists, readable, writable, parent_writable, allowed_roots, reason }`. Successful resolution inside the allowlist sets `ok:true` with the canonical `resolved` path; a `MissingPathException` (new typed subclass of `InvalidPathException`, thrown by `PathGuard::resolve()` when `realpath()` returns `false`) falls back to `PathGuard::is_writable_parent_of()` so a fresh install pointing at a not-yet-created custom path gets `ok:true` with `exists:false` and `parent_writable:true` rather than a misleading "does not exist" rejection. Branching on a typed exception (rather than the message string) means a future PathGuard message edit cannot silently break the fallback. `writable` and `parent_writable` are kept distinct on the wire so the field name doesn't lie about which inode is writable. Rejected candidates carry the PathGuard message verbatim in `reason` (`../../../etc/passwd` → "Path contains a parent-directory segment.", a `/tmp/...` outside the allowlist → "Path is outside the allowed directories."), and the configured allowlist roots ride along on every verdict so the panel can name them in error copy.
-   Settings panel "Test path" button + verdict rendering. The button calls the new endpoint via `client.testLogPath(path)`; the inline result Notice shows the resolved absolute path on success, the rejection reason on failure, and the allowlist roots in a fenced `<code>` so the admin can see what's permitted. A transport failure (network down, 5xx) lands in a separate `testError` slot rendered as a neutral warning Notice ("Could not test the path: …"), keeping the previous successful verdict (if any) visible — so the admin can tell a probe failure from a path rejection. The button auto-clears the verdict on edit, and is disabled while a probe is in flight or when the field is empty.
-   DI + plumbing: `SettingsController` now takes `PathGuard` as a second constructor argument; `Plugin::register_services()` wires the existing canonical `path_guard` service into its factory, and the new route is registered inside the same `try/catch` group that already covered `/settings` so a misconfigured allowlist leaves both routes unregistered together rather than half-registering. `InvalidPathException` is no longer `final` so `MissingPathException` can extend it.
-   Tests (Integration / `SettingsControllerTest`): existing-file inside allowlist returns `ok:true` with `resolved`/`exists:true`/`readable:true`; nonexistent-file with writable allowlisted parent returns `ok:true` with `exists:false` / `writable:false` / `parent_writable:true`; nonexistent-file whose parent is outside the allowlist returns `ok:false` (regression guard for the typed-exception fallback); `../../../etc/passwd` returns `ok:false` with a non-empty `reason` (the Phase 8.2 AC); a path outside the allowlist returns `ok:false` with a reason mentioning "outside"; an empty/whitespace path returns `ok:false` with `Path is empty.`; and `register_routes()` is asserted to register both `/settings` (GET + POST) and `/settings/test-path` (POST with a required `path` arg). Existing test sandbox cleanup drops `@unlink`/`@rmdir` suppression so a leaking handle in CI surfaces.

## [0.7.0] - 2026-04-27

Closes Phase 7 of the [roadmap](ROADMAP.md): filters, grouping, stack-trace expansion, and tail mode. The Logs tab now ships a filter bar (severity multi-select, debounced regex, date range, source dropdown) URL-mirrored via `history.replaceState`, a list ↔ grouped view toggle backed by signature grouping, click-to-copy frame paths in expanded fatals, and a tail-mode toggle that polls `GET /logs?since=<last_byte>` on the admin-configured cadence with rotation detection and an "N new entries" pill when the user has scrolled away. Backend additions: `LogQuery::$since_byte`, `PagedResult::$last_byte`/`$rotated`, `LogsController` now serialises `frames` (gated to fatal/parse severities) and surfaces `last_byte`/`rotated` on every response. Settings UI lands in Phase 8.

### Changed

-   Phase 7.3 / 7.4 review follow-ups. **Tail rotation** — `LogRepository::query()` now treats `since_byte > $last_byte` as "the file shrunk between polls" (rotation or manual clear) rather than a no-op delta. The repository falls through to the normal full-read path and `PagedResult::$rotated` (new) propagates through the REST envelope as `rotated: true`; the React store's `TAIL_APPEND_ENTRIES` reducer treats a rotated response as a baseline replacement (clears the list, drops `newCount`, prunes `expandedTraces`) instead of prepending stale-cursor zero-length deltas. Without this, a rotated log would freeze the tail UI at the rotation point until the user manually refreshed. **Filter race in tail polling** — `useTailPolling` snapshots `filtersRef.current` at request send-time and discards the response if the live shape no longer matches, so a filter change mid-flight cannot land entries that don't satisfy the active filter set. **Frame-parse cost** — `LogsController::shape_entry()` now gates `StackTraceParser::parse()` on `severity === 'fatal' || 'parse'`; warnings, notices, deprecation rows never carry traces in WP's debug-log format, and gating cuts the per-row regex sweep on the typical noisy log by ~50× without changing the wire shape. **`entryKey` collisions** — the row identifier no longer short-circuits on `entry.raw` alone (PHP error logs routinely repeat identical lines, and a `raw`-only key would expand all duplicates when one was clicked); it now composes `timestamp|file|line|severity|message|raw` and lives in `assets/src/utils/entryKey.js` so the row component and the store's expand-prune logic share one definition. **Unbounded `expandedTraces` growth** — the `LOGS_RECEIVED` reducer now prunes the map to keys present in the new payload, so a long admin session with many filter changes can't bloat memory with dead toggle keys. **Grouped-while-tailing** — flipping `viewMode` to `grouped` while tail is active now auto-stops the polling loop in the `SET_VIEW_MODE` reducer (the symmetric case to "tail-on flips to list"), so the loop can no longer keep mutating state against a list the user can't see. **`buildFilterParams`** — the filter→query-string translation lives in `assets/src/utils/filterParams.js` and is shared between `LogViewer`'s primary fetch and `useTailPolling`'s loop, so the two cannot drift in their encoding. **Action arg rename** — `appendTailEntries`'s third argument is `atTop` (matched to what the viewer actually computes); the previous `scrolledToBottom` name read backwards because the viewer renders newest-first. The bogus `total: items.length` overwrite on tail append (which clobbered the server-reported total) is removed. New PHPUnit coverage for the tail-mode backend lives in `LogQueryTest` (since_byte 0/null/negative) and `LogRepositoryTest` (since_byte 0/mid/=size/>size, last_byte propagation across list and grouped paths) plus `LogsControllerTest` cases for `frames` shape on a fatal-with-trace fixture, the Notice gate that suppresses frame parsing, the `since` round-trip, and the rotation flag on the wire — closing the test gap the review flagged.
-   `LogQuery::parse_bound()` now uses the `Y-m-d|` reset format so a date-only bound parses to midnight rather than inheriting `DateTimeImmutable::createFromFormat()`'s default of "current time of day", which made `test_date_range_filter_excludes_outside_window` flaky after noon UTC.

### Added

-   `StackTracePanel` component (Phase 7.3) — expandable panel that lists each parsed frame with its `file:line` rendered as a click-to-copy button (per the AC, the clipboard payload is the exact `path/to/file.php:123` string). Frames lacking a file (`{main}`, `[internal function]`) drop the click target since there's nothing useful to copy. Per-row expansion state lives in the `logscope/core` store as an `expandedTraces` map keyed by the entry's `raw` line, because react-window recycles row components on scroll and a row-local `useState` would lose the toggle the moment the entry passed out of the viewport. To accommodate the inline panel, `LogViewer` switches the virtualized list to a function-form `rowHeight` that closes over the `expandedTraces` map — collapsed rows stay 48px and expanded rows grow to fit their frames, and the closure changing on every toggle is what tells react-window to re-measure. `LogsController::shape_entry()` now serialises a `frames` field via `StackTraceParser::parse( $entry->raw )` so the React side has the parsed structure without re-parsing client-side; parsing happens lazily on serialise rather than at parse time, since most entries are warnings/notices that have no trace and the regex sweep would be wasted work on rows the UI will never expand. `EntryRow` now keys on the lowercase `Severity::*` tokens via the shared `severityTone`/`severityLabel` util landed in 7.1 and renders a `▾ / ⋯` toggle in the row head; the row's outer `style` from react-window still positions and sizes the row, while the trace panel renders inside.
-   Tail mode (Phase 7.4) — toolbar toggle that polls `GET /logs?since=<last_byte>` on the admin-configured `tail_interval` and prepends new entries to the current list. The polling loop is a `setTimeout` chain (rather than `setInterval`) so a slow REST call cannot pile up overlapping requests; each tick waits for the previous response before scheduling the next, which means the practical floor on a slow host is the round-trip time, not the configured interval. Auto-scroll-to-top fires when the user is within 8px of the top of the list (the viewer renders newest-first, so "scroll to where new entries land" is the top edge); when the user has scrolled away, an `aria-live="polite"` "N new entries" pill appears, and clicking it both scrolls to the top and clears the counter. Filters are honoured during tail polling — only entries that match the current severity / regex / source filter are appended — so a user watching for fatals during a deploy doesn't drown in unrelated notices. Toggling tail on while in grouped mode flips the view to list, since the `since` byte-offset endpoint returns raw entries and grouped doesn't apply. New backend surface: `LogQuery::$since_byte` (validated `>= 0`), `LogRepository::query()` honours it by reading only `[since_byte, EOF)`, skipping pagination and returning every matching entry in chronological order, and `PagedResult` carries a new `last_byte` field equal to the source size at read time so the next poll has a starting point. Even non-tail responses now expose `last_byte`, so flipping tail on after a regular fetch resumes from the right offset without an extra round-trip. `since` past EOF is a no-op (a rotated/cleared log between polls returns nothing rather than erroring), and a value ahead of the file the server can see is treated as "nothing new since you last asked." `AssetLoader::localized_payload()` now ships `tailInterval` from `Settings::get('tail_interval')` so the React loop doesn't need a settings round-trip on page load. New `useTailPolling` hook owns the loop; the LogViewer handles the scroll-to-top affordance via the `useListRef` imperative element. Existing `AssetLoader` tests were updated to construct the loader with a real `Settings` instance and a stubbed `get_option` (Settings is final, so a Mockery double can't replace it).

-   `FilterBar` component (Phase 7.1) — severity multi-select (checkbox group keyed on the lowercase `Severity::*` tokens the REST layer ships, so the wire format and the UI control share one vocabulary), `from`/`to` date inputs, a regex search debounced 300ms, and a source dropdown populated from distinct file paths in the currently-loaded items. Filter state lives in the `logscope/core` store; a new `useUrlQuerySync` hook mirrors `view`, `severity`, `from`, `to`, `q`, and `source` into the URL via `history.replaceState` (so the back stack isn't littered with one entry per keystroke) and seeds initial state from the URL on mount, satisfying the AC that any filter change updates the URL + refetches. The regex input owns its own ephemeral state and lands in the store only after the debounce, so a fast typist doesn't fire one REST query per character. The source dropdown derives in-memory from `items[].file` rather than calling a separate `distinct_sources` REST route — the data is already in the response and a per-keystroke roundtrip would be wasteful; the trade-off is that the dropdown reflects the current page's sources, not the entire log.
-   `GroupedView` component (Phase 7.2) and a list/grouped mode toggle in the Logs tab. The toggle is wired to `viewMode` in the store, which the fetch effect translates into `?grouped=true` on the existing `GET /logs` route — no new endpoint. Toggling preserves filters (they're a separate slice) and restores per-mode scroll position: react-window v2's `<List>` exposes its scroll container via the imperative `element` field on `useListRef`, so the LogViewer captures `scrollTop` on each scroll event and writes it back on remount; the grouped pane uses a plain ref on its overflow container for the same effect. Group rows render severity pill, occurrence count, file:line, sample message, and an expandable detail pane (first/last seen, location, full sample). Groups are non-virtualized: the server caps a page at 500 rows (and ships 50 by default), DOM size is bounded, and the variable-height expanded panel would otherwise demand a measuring virtualizer. The expanded view shows the group's representative sample rather than every matching entry — surfacing every entry in a signature requires server-side filtering by signature hash, which is a Phase 8 / post-1.0 backend gap; flagging here so the limitation is captured.
-   `assets/src/utils/severity.js` — single source of truth for the `Severity::*` token → CSS-tone and translated-label mapping, used by `EntryRow`, `GroupedView`, and the `FilterBar` checkbox legend. Replaces the inline map in `EntryRow` that keyed on the human strings (`'Fatal error'`, `'Warning'`) and consequently always missed (the REST layer serialises lowercase tokens), so every row had previously rendered with the unknown-tone styling. `useDebouncedValue` and `useUrlQuerySync` hooks land alongside as small, single-purpose helpers.

## [0.6.0] - 2026-04-27

Closes Phase 6 of the [roadmap](ROADMAP.md): the admin page + React viewer shell. Logscope is now a visible feature in wp-admin — **Tools → Logscope** renders a React app gated by `logscope_manage`, served by a screen-scoped asset pipeline and a `@wordpress/data` store consuming the Phase-4/5 REST surface. Filters, grouping, trace expansion, and tail mode arrive in Phase 7; the settings UI lands in Phase 8.

### Added

-   `LogViewer`, `EntryRow`, `EmptyState` React components — the Logs tab now renders a virtualized list of debug-log entries fetched from `GET /logs`. Virtualization is `react-window` v2's `<List>` (added as a dep — ~6 KB gz, far less risk than a hand-rolled virtualizer); the rendered DOM is bounded by viewport height regardless of result-set size. Row height is fixed at 48px for the Phase 6 shell — variable heights would push trace expansion into a second virtualizer and double the per-row measure cost; we'll revisit if 7.3 reveals a UX problem with the truncated message. `EntryRow` renders a severity pill (mapped to fatal/warning/notice/deprecated/unknown tones), the WP timestamp, the truncated message, and a small `⋯` indicator on entries that carry a stack trace; the actual trace expansion is intentionally deferred to `StackTracePanel` in step 7.3 because per-row React state is lost when react-window recycles the row component on scroll, so the right home for the expanded/collapsed map is a parent-owned dictionary keyed by signature. `EmptyState` distinguishes a fresh-install zero-results screen from a fetch-failure error screen, so an admin sees a different cue when the network blew up versus when the log is genuinely empty. `LogViewer` dispatches `fetchLogs` on mount and renders a `Spinner` only on the first load (subsequent refreshes show the existing list with `aria-busy=true` so the user isn't punished for refetching). Bundle weight grew from 3.26 KiB to 14 KiB on this step (`react-window` + the three new components). The 5k-entry smoke test the AC calls for has not been run yet — it requires a live WP install to exercise; deferring to the human-eyes pass before the v0.6.0 tag in 6.5.
-   React app shell (`assets/src/index.js`, `components/App.jsx`, `store/index.js`, `api/client.js`). Entry mounts on `#logscope-root` (the contract `PageRenderer::ROOT_ELEMENT_ID` mirrors), gated on `DOMContentLoaded` so a script ordering quirk on the host page can't race the mount. The `@wordpress/data` store is registered as `logscope/core` with a minimal Phase-6 shape (active tab + a logs slot the viewer in 6.4 populates); the `fetchLogs` generator yields each action (start → receive | fail) rather than returning the terminal action, since `@wordpress/redux-routine` only dispatches yielded values — a returned action is the generator's return value, not a dispatch, and would leave the store stuck in `isLoading: true`. The REST client wraps `@wordpress/api-fetch` with `createRootURLMiddleware` (REST root) + `createNonceMiddleware` (the `wp_rest` nonce), both read from `window.LogscopeAdmin` that `AssetLoader` localizes — every Logscope call goes through one client so the nonce + namespace contract has a single owner. Middleware registration is guarded by a flag stashed on `apiFetch` itself so an HMR re-import or a duplicate import path can't stack the same middleware twice (which would prepend the REST root to an already-prefixed URL). The shell renders a two-tab layout (Logs · Settings) controlled by the store rather than `<TabPanel initialTabName>` (the latter only seeds once and would not repaint on a back-button hash navigation); a single hashchange listener turns every hash mutation into one store dispatch, and tab clicks set `location.hash` so the listener is the single source of truth — eliminating the double-dispatch the earlier draft had. Tab bodies render the LogViewer (Logs) or a placeholder pointing at Phase 8 (Settings).
-   `Logscope\Admin\AssetLoader` — enqueues the React bundle (`assets/build/index.js` + optional `index.css`) **only on the Logscope screen**, comparing the `admin_enqueue_scripts` hook suffix against the value `Menu::register()` captured at registration time so unrelated wp-admin pages stay byte-for-byte identical (a recurring wp.org reviewer flag for plugins that enqueue everywhere). Dependencies and version hash come from the `index.asset.php` file `@wordpress/scripts` writes on every successful build, so adding a `@wordpress/components` import to a React file automatically wires `wp-components` into the enqueue without hand-maintaining a dep list. A missing asset file is treated as "build hasn't run / broken zip" and silently skips enqueue; the renderer's empty `#logscope-root` then serves as the visible failure mode. CSS is conditionally enqueued only when `index.css` exists, since pure-JS builds don't emit a stylesheet. `wp_localize_script` writes a `window.LogscopeAdmin` bootstrap object carrying `restUrl` (Logscope namespace), `restRoot` (for `@wordpress/api-fetch`), `nonce`, and `canManage`; `wp_set_script_translations` wires up the `logscope` text domain. `Plugin` registers `admin.asset_loader` and hooks `enqueue_admin_assets` on `admin_enqueue_scripts` inside the same `Throwable` trap as the other lifecycle callbacks. Unit tests cover the screen-id no-op, the missing-asset-file no-op, the script enqueue with deps + version, the conditional CSS enqueue, and the localized payload shape.
-   `Logscope\Admin\Menu` and `Logscope\Admin\PageRenderer` — register the **Tools → Logscope** submenu (slug `logscope`) gated by the `logscope_manage` capability (resolved through `Capabilities::required()` so the `logscope/required_capability` filter still applies). The renderer outputs the wp-admin `<div class="wrap">` shell with a translated `<h1>Logscope</h1>` and a single mount node (`<div id="logscope-root">`) the React app will hydrate in step 6.3 — kept intentionally bare so a JS bundle failure surfaces as a visibly empty page rather than a half-rendered hybrid. `PageRenderer::ROOT_ELEMENT_ID` is the canonical contract between PHP and the React entry. `Plugin` registers `admin.page_renderer` + `admin.menu` services and wires `register_admin_menu()` on `admin_menu`, wrapped in the same `Throwable` trap as `register_rest_routes()` so a DI-graph failure routes through the `WP_DEBUG`-gated breadcrumb instead of aborting menu registration for unrelated plugins. Unit tests cover the `add_submenu_page` invocation shape, the filter override path, the `false` return defensiveness, and the renderer markup contract.

## [0.5.0] - 2026-04-27

Closes Phases 4 and 5 of the [roadmap](ROADMAP.md): the REST surface React will consume in Phase 6 (`GET/DELETE /logs`, `GET /logs/download`, `GET/POST /settings`) plus the schema-driven settings backend. The `RestController` abstract base centralises capability + authentication enforcement so every Logscope route inherits a uniform 401/403 contract; the `Plugin` DI graph wires `path_guard` → `log_source` → `log_repository` → controllers and registers logs and settings routes in independent `try/catch` blocks so a misconfiguration in one cannot abort `rest_api_init` for the other or for unrelated plugins. Still no user-visible features — the React admin page lands in Phase 6.

### Changed

-   Phase 5 review follow-ups. `Plugin::register_rest_routes()` no longer silently swallows DI-graph failures: each `try/catch` now routes through a private `log_route_registration_failure()` helper that writes a one-line breadcrumb (route group, exception class, message) via `error_log()` when `WP_DEBUG` is on, so an admin reporting a 404 on `/logs` or `/settings` has something to grep for. `SettingsController::handle_post()` now applies writes in two phases — sanitise the entire body into a temp array first, then persist — so a sanitiser that throws on a later key cannot leave earlier keys partially saved in `wp_options`. The previously-dead `logscope_rest_invalid_setting` catch is now reachable for any future schema-side validation. `SettingsSchema` builds its field map eagerly in the constructor (the lazy guard bought nothing, since closures were already `static` and the data is constant) and exposes a `TYPES` const that closes the type vocabulary; `matches_type()` no longer needs a defensive default branch. The `tail_interval` sanitiser now uses `is_numeric()` + `(int)` so `'3.5'` truncates to `3` instead of falling back to the default `1`, matching what an admin would expect from a number input. `SettingsSchema` is no longer `final` so a test-only subclass can inject a sanitiser that throws — used by a new atomicity test that proves a phase-1 failure on `tail_interval` does not leak a phase-2 write of `log_path`.
-   Phase 4 review follow-ups. Removed the unused `RestController::register()` indirection — `Plugin::register_rest_routes()` is now the single registration point for Logscope routes, eliminating a latent double-registration hazard. `Plugin::register_rest_routes()` now traps `Throwable` (was `InvalidPathException` only) so any constructor-time failure in the `path_guard` → `log_source` → `log_repository` → `rest.logs_controller` graph leaves the routes unregistered without aborting `rest_api_init` for other plugins. `LogsController::normalise_severity_param()` now drops unknown tokens against `Severity::all()` for the comma-separated string path, matching the args-schema enum constraint that already covered the array form. `LogsController::download_headers_for()` strips control chars, double-quotes, and backslashes from the basename before interpolating into `Content-Disposition`, falling back to `debug.log` if everything is stripped, so a sibling renamed externally cannot produce a malformed header. The `_logscope_skip_exit_for_tests` request-param escape hatch now carries an explicit `@internal` docblock noting it must be removed if the cap gate ever weakens.

### Added

-   `Logscope\REST\SettingsController` — `GET` + `POST /wp-json/logscope/v1/settings`. `GET` returns the full settings shape produced by `Settings::all()`; `POST` accepts a partial body, rejects bodies that contain any key the schema does not know with a 400 `logscope_rest_unknown_setting` (the error data carries the offending keys under `unknown` so the React client can highlight the right field), then sanitises and persists each known key through `Settings::set()` and returns the new full shape so the client can update its store without a follow-up GET. The unknown-key rejection is all-or-nothing: a request mixing one valid and one invalid key is refused outright rather than partially saved, so a typo cannot leave the settings in a half-applied state. `InvalidArgumentException` from the schema is caught and translated to a 400 `logscope_rest_invalid_setting` as a belt-and-suspenders fallback. `Plugin` registers `rest.settings_controller` in the DI container, and `register_rest_routes()` now wires logs and settings routes in independent `try/catch` blocks so a failure in one DI subgraph cannot take down the other (or abort `rest_api_init` for unrelated plugins). Integration tests (with a `wp_options`-backed `get_option`/`update_option` stub) cover the GET shape, POST persisting + returning new state with sanitisation, the AC `tail_interval=0` → `1` coercion, the unknown-key 400 (with `unknown` payload + no partial save), the empty-body no-op, and the `register_rest_route()` invocation shape.
-   `Logscope\Settings\SettingsSchema` and `Logscope\Settings\Settings` — schema-driven settings facade backing the v1.0 surface (`log_path`, `tail_interval`). The schema is the single source of truth for each field's underlying `wp_options` row name, scalar type, default, and sanitizer; a sanitizer always runs on `set()` so the REST layer can pass raw input through without pre-cleaning. `tail_interval` coerces non-int input and values `< 1` to `1` (matching the roadmap "min 1s"); `log_path` strips null bytes and trims whitespace, leaving filesystem-semantic validation to `PathGuard` on use. `Settings::get()` re-validates the stored value against the declared type and falls back to the schema default when an older version of the plugin left a corrupted row, so callers never see the wrong shape. `Settings::all()` exposes the full settings view that the REST controller will return on `GET /settings`. `Settings::set()` and the schema both throw `InvalidArgumentException` for unknown keys; the REST boundary translates that into a 400. `Plugin` registers `settings.schema` and `settings` services in the DI container, and `Activator::DEFAULT_OPTIONS` carries a docblock cross-link so the activation seed and the runtime schema cannot drift unnoticed. Unit tests cover the schema (defaults, sanitizer happy paths, `tail_interval=0` → `1`, type matchers, unknown-key rejection) and the facade (Brain-Monkey-mocked `get_option`/`update_option` round-trip, wrong-type fallback, set-strips-null-bytes, `all()` shape).
-   `Logscope\REST\RestController` — abstract base for every Logscope REST controller. Centralises the `logscope/v1` namespace, `register_routes()` registration on `rest_api_init`, a uniform `permission_callback()` returning a 401 `WP_Error` for unauthenticated callers and a 403 for users missing the Logscope manage capability, and an `error()` factory that maps internal failures to `WP_Error` shapes the REST dispatcher will turn into the right HTTP status. Cookie-nonce verification stays delegated to core's `rest_authentication_errors` pipeline so application-password and other auth schemes plug in without bespoke handling. Unit tests cover the 401/403/allowed branches, the `rest_api_init` registration, the `error()` payload shape, and the namespace constant; a slim `WP_Error` stub plus a stub subclass make the base testable without a full WordPress harness.
-   `LogsController` — `DELETE /wp-json/logscope/v1/logs` and `GET /wp-json/logscope/v1/logs/download`. Clear is a soft-delete that renames the active log to `<basename>.cleared-YYYYMMDD-HHMMSS` (UTC) so a wiped log is still recoverable for post-mortems; the destructive verb requires `?confirm=true` to fire (returns a 400 `logscope_rest_confirmation_required` otherwise) and 404s when no log file exists yet. Before renaming, the controller asks `PathGuard::is_writable_parent_of()` to confirm the directory is writable, returning a 403 `logscope_rest_log_not_writable` when it isn't, so a misconfigured `chown` surfaces as a clean error rather than a silent failure. The download route streams the raw log with `Content-Type: text/plain`, `Content-Disposition: attachment`, `X-Content-Type-Options: nosniff`, an explicit `Content-Length`, and `Cache-Control: no-store` headers, then `readfile()` + `exit` to bypass core's REST JSON serialiser; a `_logscope_skip_exit_for_tests` request param skips header-emit and `exit` so the unit suite can drive the streaming path under PHPUnit's already-flushed output buffer. `LogsController::download_headers_for()` and `archive_path_for()` are exposed as static pure helpers so the wire contract is asserted directly without filesystem or networking. New tests cover: clear-without-confirm 400, clear-on-missing-log 404, clear renames file with the expected `cleared-YYYYMMDD-HHMMSS` pattern, archive-path helper, download-on-missing-log 404, download header shape, and end-to-end stream-to-output-buffer.
-   `Logscope\Support\PathGuard::is_writable_parent_of()` — companion to `is_writable()` that resolves the parent directory of a candidate path through the existing allowlist and returns `true` only when the parent is canonical, allowlisted, and writable by the PHP process. Used by `LogsController::handle_clear` for the soft-delete pre-flight and earmarked for the Phase 8 "Test path" settings probe; the file itself need not exist yet, which matches both use cases. Unit tests cover the writable-parent happy path, outside-allowlist rejection, null-byte rejection, and empty-string rejection.
-   `Logscope\REST\LogsController` — `GET /wp-json/logscope/v1/logs` returning paginated, filtered log views. Args schema accepts `page`, `per_page` (1–500, default 50), `severity` (array of `Severity` enum values, also accepts a comma-separated string for clients that prefer flat query strings), `from` / `to`, `q` (regex, ≤200 chars), `grouped`, and `source`; core's REST validator rejects out-of-range or wrong-type params before our handler runs. Response body is `{items, total, page, per_page, total_pages}`, and pagination totals are mirrored into `X-WP-Total` / `X-WP-TotalPages` headers so the React client matches core's collection convention. `LogQueryException` (bad regex, unparseable date, etc.) is caught and translated to a 400 `WP_Error` rather than bubbling as a 500. `Plugin` now owns a small DI graph wiring `path_guard` → `log_source` → `log_repository` → `rest.logs_controller`, and registers the controller's routes on `rest_api_init`; a misconfigured custom log path causes `register_rest_routes()` to swallow the `InvalidPathException` so other plugins' route registration is not aborted. Tests stub `WP_REST_Request` / `WP_REST_Response` (via test bootstrap) and exercise the handler end-to-end against a real fixture log: the AC scenario (page 2 of 50 with `severity=Fatal` over 75 fatals) plus grouped output, comma-separated severity, missing-log empty set, the 400-on-bad-regex path, and the `register_rest_route()` invocation shape.

## [0.4.0] - 2026-04-27

Closes Phase 3 of the [roadmap](ROADMAP.md): the log-reading and parsing foundation. Logscope can now safely locate a `debug.log` (PathGuard allowlist + traversal protection), stream bytes out of it without loading the whole file into memory, parse the WordPress severity formats and PHP stack traces, group "the same error happening over and over" by signature, and serve filtered, paginated views through a single repository facade. Still no user-visible features — REST routes and the React UI come next in Phase 4 onward.

### Added

-   `Logscope\Support\PathGuard` — filesystem path validator with two-stage rejection (raw-string checks for null bytes and `..` segments before any filesystem call, then `realpath()` canonicalisation with allowlist containment). Symlink escapes are caught implicitly because `realpath()` resolves links before the containment check; sibling-prefix attacks (e.g. `/var/www-evil` against root `/var/www`) are rejected by the exact-separator suffix match. Constructor fails closed: a non-existent root is silently dropped so a misconfigured root cannot widen the allowlist. Companion `InvalidPathException` carries plain-English internal messages; translation happens at the REST/UI boundary later.
-   `Logscope\Log\LogSourceInterface` and `FileLogSource` — byte-level abstraction over the underlying log file with `path() / exists() / size() / read_chunk(from, max)`. Reads use `fopen('rb') / fseek / fread` so multi-hundred-MB logs never hit memory all at once. Constructor validates the parent directory through `PathGuard` and reassembles the final path from the validated dirname plus `basename()` of the input — keeping the contract usable on a fresh WordPress install where `debug.log` does not yet exist, while still defeating traversal in the input. Past-EOF, missing-file, and non-positive-bound reads return an empty string rather than throwing.
-   `Logscope\Log\LogParser`, `Entry`, and `Severity` — pure parser turning raw debug-log text into structured `Entry[]`. Recognises all six WordPress severity tokens (`Fatal error`, `Parse error`, `Warning`, `Notice`, `Deprecated`, `Strict Standards`) plus an `unknown` bucket for unsevered `error_log()` output. Parses WP's `[DD-Mon-YYYY HH:MM:SS [TZ]]` timestamp form with or without the timezone token. Continuation lines (stack-trace rows, `thrown in ...` tails, blank separators) attach to the previous entry's `raw` field; orphan continuations at chunk start are dropped because chunk stitching is the future repository's responsibility. File and line are extracted from both `... in /path.php:N` and `... in /path.php on line N` shapes.
-   `Logscope\Log\StackTraceParser` and `Frame` — parses PHP's `#N <file>(<line>): <call>` stack-trace rows into `Frame[]`. Recognises bare-function, instance (`->`), and static (`::`) calls with namespaced class names; special `[internal function]` and `{main}` frames are represented by null patterns rather than dedicated subtypes. The file regex anchors on the rightmost `(\d+):` so Windows-style paths like `C:\Program Files (x86)\app\main.php` parse correctly even with parens earlier in the path. Argument lists are captured as raw strings and never evaluated, since PHP's trace output for arguments routinely contains truncated literals and `Object(...)` placeholders that are not valid source.
-   `Logscope\Log\LogGrouper` and `Group` — collapses `Entry[]` into "the same error happening over and over" buckets. The signature is `md5(severity | file | line | normalised-message)`, where normalisation strips hex addresses (`0x[0-9a-fA-F]+` → `0xN`), single- and double-quoted strings (→ `'?'` / `"?"`), and bare integers (→ `N`). Order matters: hex first so the digit rule does not eat address bytes. Groups carry a running `count`, the first-observed `sample_message` for display, and a `first_seen` / `last_seen` window built from parsed `DateTimeImmutable` comparisons rather than raw string compares (the WP timestamp format is not lexically sortable across months). Output is sorted by descending count, with ties broken by most-recent `last_seen` then signature for stable ordering across runs.
-   `Logscope\Log\LogRepository`, `LogQuery`, `PagedResult`, `SourceClassifier`, and `LogQueryException` — facade over the source / parser / grouper stack with paginated, filtered access. `LogQuery` validates severity, date window (`Y-m-d` or `Y-m-d H:i:s`), regex (≤ 200 chars, server-side, message-scoped), source slug (`plugins/<slug>` / `themes/<slug>` / `mu-plugins/<slug>` / `core` produced by `SourceClassifier`), grouped flag, and pagination at construction; downstream code can trust every field. `LogRepository::query()` returns a `PagedResult` with `items`, `total`, `page`, `per_page`, `total_pages` ready for `X-WP-Total` / `X-WP-TotalPages` headers. Ungrouped queries return entries newest-first (reverse of file order, since WP appends); grouped queries return `Group[]` in count-desc order. Whole-file reads are bounded by `MAX_BYTES_PER_QUERY = 50 MB` — pathological logs are tail-clipped and the parser drops the orphan continuation at the cutoff. `distinct_sources()` exposes the dropdown population list. New `Integration` PHPUnit suite registered; `tests/php/Integration/Log/LogRepositoryTest.php` covers the AC end-to-end (page 2 of 50 with `severity=Fatal` over a 75-fatal fixture).

### Security

-   PathGuard is the foundation for all log-file access in later phases — every `FileLogSource` and settings-side path test will route through it before touching disk.
