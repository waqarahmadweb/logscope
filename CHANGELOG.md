## [Unreleased]

### Added

-   `Logscope\Admin\AssetLoader` â€” enqueues the React bundle (`assets/build/index.js` + optional `index.css`) **only on the Logscope screen**, comparing the `admin_enqueue_scripts` hook suffix against the value `Menu::register()` captured at registration time so unrelated wp-admin pages stay byte-for-byte identical (a recurring wp.org reviewer flag for plugins that enqueue everywhere). Dependencies and version hash come from the `index.asset.php` file `@wordpress/scripts` writes on every successful build, so adding a `@wordpress/components` import to a React file automatically wires `wp-components` into the enqueue without hand-maintaining a dep list. Unit tests cover the screen-id no-op, the missing-asset-file no-op, the script enqueue with deps + version, the conditional CSS enqueue, and the localized payload shape.
-   `Logscope\Admin\Menu` and `Logscope\Admin\PageRenderer` â€” register the **Tools â†’ Logscope** submenu (slug `logscope`) gated by the `logscope_manage` capability (resolved through `Capabilities::required()` so the `logscope/required_capability` filter still applies). The renderer outputs the wp-admin `<div class="wrap">` shell with a translated `<h1>Logscope</h1>` and a single mount node (`<div id="logscope-root">`) the React app will hydrate in step 6.3 â€” kept intentionally bare so a JS bundle failure surfaces as a visibly empty page rather than a half-rendered hybrid. `PageRenderer::ROOT_ELEMENT_ID` is the canonical contract between PHP and the React entry. `Plugin` registers `admin.page_renderer` + `admin.menu` services and wires `register_admin_menu()` on `admin_menu`, wrapped in the same `Throwable` trap as `register_rest_routes()` so a DI-graph failure routes through the `WP_DEBUG`-gated breadcrumb instead of aborting menu registration for unrelated plugins. Unit tests cover the `add_submenu_page` invocation shape, the filter override path, the `false` return defensiveness, and the renderer markup contract.

## [0.5.0] - 2026-04-27

Closes Phases 4 and 5 of the [roadmap](ROADMAP.md): the REST surface React will consume in Phase 6 (`GET/DELETE /logs`, `GET /logs/download`, `GET/POST /settings`) plus the schema-driven settings backend. The `RestController` abstract base centralises capability + authentication enforcement so every Logscope route inherits a uniform 401/403 contract; the `Plugin` DI graph wires `path_guard` â†’ `log_source` â†’ `log_repository` â†’ controllers and registers logs and settings routes in independent `try/catch` blocks so a misconfiguration in one cannot abort `rest_api_init` for the other or for unrelated plugins. Still no user-visible features â€” the React admin page lands in Phase 6.

### Changed

-   Phase 5 review follow-ups. `Plugin::register_rest_routes()` no longer silently swallows DI-graph failures: each `try/catch` now routes through a private `log_route_registration_failure()` helper that writes a one-line breadcrumb (route group, exception class, message) via `error_log()` when `WP_DEBUG` is on, so an admin reporting a 404 on `/logs` or `/settings` has something to grep for. `SettingsController::handle_post()` now applies writes in two phases â€” sanitise the entire body into a temp array first, then persist â€” so a sanitiser that throws on a later key cannot leave earlier keys partially saved in `wp_options`. The previously-dead `logscope_rest_invalid_setting` catch is now reachable for any future schema-side validation. `SettingsSchema` builds its field map eagerly in the constructor (the lazy guard bought nothing, since closures were already `static` and the data is constant) and exposes a `TYPES` const that closes the type vocabulary; `matches_type()` no longer needs a defensive default branch. The `tail_interval` sanitiser now uses `is_numeric()` + `(int)` so `'3.5'` truncates to `3` instead of falling back to the default `1`, matching what an admin would expect from a number input. `SettingsSchema` is no longer `final` so a test-only subclass can inject a sanitiser that throws â€” used by a new atomicity test that proves a phase-1 failure on `tail_interval` does not leak a phase-2 write of `log_path`.
-   Phase 4 review follow-ups. Removed the unused `RestController::register()` indirection â€” `Plugin::register_rest_routes()` is now the single registration point for Logscope routes, eliminating a latent double-registration hazard. `Plugin::register_rest_routes()` now traps `Throwable` (was `InvalidPathException` only) so any constructor-time failure in the `path_guard` â†’ `log_source` â†’ `log_repository` â†’ `rest.logs_controller` graph leaves the routes unregistered without aborting `rest_api_init` for other plugins. `LogsController::normalise_severity_param()` now drops unknown tokens against `Severity::all()` for the comma-separated string path, matching the args-schema enum constraint that already covered the array form. `LogsController::download_headers_for()` strips control chars, double-quotes, and backslashes from the basename before interpolating into `Content-Disposition`, falling back to `debug.log` if everything is stripped, so a sibling renamed externally cannot produce a malformed header. The `_logscope_skip_exit_for_tests` request-param escape hatch now carries an explicit `@internal` docblock noting it must be removed if the cap gate ever weakens.

### Added

-   `Logscope\REST\SettingsController` â€” `GET` + `POST /wp-json/logscope/v1/settings`. `GET` returns the full settings shape produced by `Settings::all()`; `POST` accepts a partial body, rejects bodies that contain any key the schema does not know with a 400 `logscope_rest_unknown_setting` (the error data carries the offending keys under `unknown` so the React client can highlight the right field), then sanitises and persists each known key through `Settings::set()` and returns the new full shape so the client can update its store without a follow-up GET. The unknown-key rejection is all-or-nothing: a request mixing one valid and one invalid key is refused outright rather than partially saved, so a typo cannot leave the settings in a half-applied state. `InvalidArgumentException` from the schema is caught and translated to a 400 `logscope_rest_invalid_setting` as a belt-and-suspenders fallback. `Plugin` registers `rest.settings_controller` in the DI container, and `register_rest_routes()` now wires logs and settings routes in independent `try/catch` blocks so a failure in one DI subgraph cannot take down the other (or abort `rest_api_init` for unrelated plugins). Integration tests (with a `wp_options`-backed `get_option`/`update_option` stub) cover the GET shape, POST persisting + returning new state with sanitisation, the AC `tail_interval=0` â†’ `1` coercion, the unknown-key 400 (with `unknown` payload + no partial save), the empty-body no-op, and the `register_rest_route()` invocation shape.
-   `Logscope\Settings\SettingsSchema` and `Logscope\Settings\Settings` â€” schema-driven settings facade backing the v1.0 surface (`log_path`, `tail_interval`). The schema is the single source of truth for each field's underlying `wp_options` row name, scalar type, default, and sanitizer; a sanitizer always runs on `set()` so the REST layer can pass raw input through without pre-cleaning. `tail_interval` coerces non-int input and values `< 1` to `1` (matching the roadmap "min 1s"); `log_path` strips null bytes and trims whitespace, leaving filesystem-semantic validation to `PathGuard` on use. `Settings::get()` re-validates the stored value against the declared type and falls back to the schema default when an older version of the plugin left a corrupted row, so callers never see the wrong shape. `Settings::all()` exposes the full settings view that the REST controller will return on `GET /settings`. `Settings::set()` and the schema both throw `InvalidArgumentException` for unknown keys; the REST boundary translates that into a 400. `Plugin` registers `settings.schema` and `settings` services in the DI container, and `Activator::DEFAULT_OPTIONS` carries a docblock cross-link so the activation seed and the runtime schema cannot drift unnoticed. Unit tests cover the schema (defaults, sanitizer happy paths, `tail_interval=0` â†’ `1`, type matchers, unknown-key rejection) and the facade (Brain-Monkey-mocked `get_option`/`update_option` round-trip, wrong-type fallback, set-strips-null-bytes, `all()` shape).
-   `Logscope\REST\RestController` â€” abstract base for every Logscope REST controller. Centralises the `logscope/v1` namespace, `register_routes()` registration on `rest_api_init`, a uniform `permission_callback()` returning a 401 `WP_Error` for unauthenticated callers and a 403 for users missing the Logscope manage capability, and an `error()` factory that maps internal failures to `WP_Error` shapes the REST dispatcher will turn into the right HTTP status. Cookie-nonce verification stays delegated to core's `rest_authentication_errors` pipeline so application-password and other auth schemes plug in without bespoke handling. Unit tests cover the 401/403/allowed branches, the `rest_api_init` registration, the `error()` payload shape, and the namespace constant; a slim `WP_Error` stub plus a stub subclass make the base testable without a full WordPress harness.
-   `LogsController` â€” `DELETE /wp-json/logscope/v1/logs` and `GET /wp-json/logscope/v1/logs/download`. Clear is a soft-delete that renames the active log to `<basename>.cleared-YYYYMMDD-HHMMSS` (UTC) so a wiped log is still recoverable for post-mortems; the destructive verb requires `?confirm=true` to fire (returns a 400 `logscope_rest_confirmation_required` otherwise) and 404s when no log file exists yet. Before renaming, the controller asks `PathGuard::is_writable_parent_of()` to confirm the directory is writable, returning a 403 `logscope_rest_log_not_writable` when it isn't, so a misconfigured `chown` surfaces as a clean error rather than a silent failure. The download route streams the raw log with `Content-Type: text/plain`, `Content-Disposition: attachment`, `X-Content-Type-Options: nosniff`, an explicit `Content-Length`, and `Cache-Control: no-store` headers, then `readfile()` + `exit` to bypass core's REST JSON serialiser; a `_logscope_skip_exit_for_tests` request param skips header-emit and `exit` so the unit suite can drive the streaming path under PHPUnit's already-flushed output buffer. `LogsController::download_headers_for()` and `archive_path_for()` are exposed as static pure helpers so the wire contract is asserted directly without filesystem or networking. New tests cover: clear-without-confirm 400, clear-on-missing-log 404, clear renames file with the expected `cleared-YYYYMMDD-HHMMSS` pattern, archive-path helper, download-on-missing-log 404, download header shape, and end-to-end stream-to-output-buffer.
-   `Logscope\Support\PathGuard::is_writable_parent_of()` â€” companion to `is_writable()` that resolves the parent directory of a candidate path through the existing allowlist and returns `true` only when the parent is canonical, allowlisted, and writable by the PHP process. Used by `LogsController::handle_clear` for the soft-delete pre-flight and earmarked for the Phase 8 "Test path" settings probe; the file itself need not exist yet, which matches both use cases. Unit tests cover the writable-parent happy path, outside-allowlist rejection, null-byte rejection, and empty-string rejection.
-   `Logscope\REST\LogsController` â€” `GET /wp-json/logscope/v1/logs` returning paginated, filtered log views. Args schema accepts `page`, `per_page` (1â€“500, default 50), `severity` (array of `Severity` enum values, also accepts a comma-separated string for clients that prefer flat query strings), `from` / `to`, `q` (regex, â‰¤200 chars), `grouped`, and `source`; core's REST validator rejects out-of-range or wrong-type params before our handler runs. Response body is `{items, total, page, per_page, total_pages}`, and pagination totals are mirrored into `X-WP-Total` / `X-WP-TotalPages` headers so the React client matches core's collection convention. `LogQueryException` (bad regex, unparseable date, etc.) is caught and translated to a 400 `WP_Error` rather than bubbling as a 500. `Plugin` now owns a small DI graph wiring `path_guard` â†’ `log_source` â†’ `log_repository` â†’ `rest.logs_controller`, and registers the controller's routes on `rest_api_init`; a misconfigured custom log path causes `register_rest_routes()` to swallow the `InvalidPathException` so other plugins' route registration is not aborted. Tests stub `WP_REST_Request` / `WP_REST_Response` (via test bootstrap) and exercise the handler end-to-end against a real fixture log: the AC scenario (page 2 of 50 with `severity=Fatal` over 75 fatals) plus grouped output, comma-separated severity, missing-log empty set, the 400-on-bad-regex path, and the `register_rest_route()` invocation shape.

## [0.4.0] - 2026-04-27

Closes Phase 3 of the [roadmap](ROADMAP.md): the log-reading and parsing foundation. Logscope can now safely locate a `debug.log` (PathGuard allowlist + traversal protection), stream bytes out of it without loading the whole file into memory, parse the WordPress severity formats and PHP stack traces, group "the same error happening over and over" by signature, and serve filtered, paginated views through a single repository facade. Still no user-visible features â€” REST routes and the React UI come next in Phase 4 onward.

### Added

-   `Logscope\Support\PathGuard` â€” filesystem path validator with two-stage rejection (raw-string checks for null bytes and `..` segments before any filesystem call, then `realpath()` canonicalisation with allowlist containment). Symlink escapes are caught implicitly because `realpath()` resolves links before the containment check; sibling-prefix attacks (e.g. `/var/www-evil` against root `/var/www`) are rejected by the exact-separator suffix match. Constructor fails closed: a non-existent root is silently dropped so a misconfigured root cannot widen the allowlist. Companion `InvalidPathException` carries plain-English internal messages; translation happens at the REST/UI boundary later.
-   `Logscope\Log\LogSourceInterface` and `FileLogSource` â€” byte-level abstraction over the underlying log file with `path() / exists() / size() / read_chunk(from, max)`. Reads use `fopen('rb') / fseek / fread` so multi-hundred-MB logs never hit memory all at once. Constructor validates the parent directory through `PathGuard` and reassembles the final path from the validated dirname plus `basename()` of the input â€” keeping the contract usable on a fresh WordPress install where `debug.log` does not yet exist, while still defeating traversal in the input. Past-EOF, missing-file, and non-positive-bound reads return an empty string rather than throwing.
-   `Logscope\Log\LogParser`, `Entry`, and `Severity` â€” pure parser turning raw debug-log text into structured `Entry[]`. Recognises all six WordPress severity tokens (`Fatal error`, `Parse error`, `Warning`, `Notice`, `Deprecated`, `Strict Standards`) plus an `unknown` bucket for unsevered `error_log()` output. Parses WP's `[DD-Mon-YYYY HH:MM:SS [TZ]]` timestamp form with or without the timezone token. Continuation lines (stack-trace rows, `thrown in ...` tails, blank separators) attach to the previous entry's `raw` field; orphan continuations at chunk start are dropped because chunk stitching is the future repository's responsibility. File and line are extracted from both `... in /path.php:N` and `... in /path.php on line N` shapes.
-   `Logscope\Log\StackTraceParser` and `Frame` â€” parses PHP's `#N <file>(<line>): <call>` stack-trace rows into `Frame[]`. Recognises bare-function, instance (`->`), and static (`::`) calls with namespaced class names; special `[internal function]` and `{main}` frames are represented by null patterns rather than dedicated subtypes. The file regex anchors on the rightmost `(\d+):` so Windows-style paths like `C:\Program Files (x86)\app\main.php` parse correctly even with parens earlier in the path. Argument lists are captured as raw strings and never evaluated, since PHP's trace output for arguments routinely contains truncated literals and `Object(...)` placeholders that are not valid source.
-   `Logscope\Log\LogGrouper` and `Group` â€” collapses `Entry[]` into "the same error happening over and over" buckets. The signature is `md5(severity | file | line | normalised-message)`, where normalisation strips hex addresses (`0x[0-9a-fA-F]+` â†’ `0xN`), single- and double-quoted strings (â†’ `'?'` / `"?"`), and bare integers (â†’ `N`). Order matters: hex first so the digit rule does not eat address bytes. Groups carry a running `count`, the first-observed `sample_message` for display, and a `first_seen` / `last_seen` window built from parsed `DateTimeImmutable` comparisons rather than raw string compares (the WP timestamp format is not lexically sortable across months). Output is sorted by descending count, with ties broken by most-recent `last_seen` then signature for stable ordering across runs.
-   `Logscope\Log\LogRepository`, `LogQuery`, `PagedResult`, `SourceClassifier`, and `LogQueryException` â€” facade over the source / parser / grouper stack with paginated, filtered access. `LogQuery` validates severity, date window (`Y-m-d` or `Y-m-d H:i:s`), regex (â‰¤ 200 chars, server-side, message-scoped), source slug (`plugins/<slug>` / `themes/<slug>` / `mu-plugins/<slug>` / `core` produced by `SourceClassifier`), grouped flag, and pagination at construction; downstream code can trust every field. `LogRepository::query()` returns a `PagedResult` with `items`, `total`, `page`, `per_page`, `total_pages` ready for `X-WP-Total` / `X-WP-TotalPages` headers. Ungrouped queries return entries newest-first (reverse of file order, since WP appends); grouped queries return `Group[]` in count-desc order. Whole-file reads are bounded by `MAX_BYTES_PER_QUERY = 50 MB` â€” pathological logs are tail-clipped and the parser drops the orphan continuation at the cutoff. `distinct_sources()` exposes the dropdown population list. New `Integration` PHPUnit suite registered; `tests/php/Integration/Log/LogRepositoryTest.php` covers the AC end-to-end (page 2 of 50 with `severity=Fatal` over a 75-fatal fixture).

### Security

-   PathGuard is the foundation for all log-file access in later phases â€” every `FileLogSource` and settings-side path test will route through it before touching disk.

## [0.3.0] - 2026-04-23

Closes Phase 2 of the [roadmap](ROADMAP.md): the plugin now has a bootstrap entry point, a lightweight DI container, activation/deactivation/uninstall lifecycle, a reusable capability helper, and the PHPUnit + Brain Monkey scaffolding future tests will build on. Still no user-visible features.

### Added

-   Plugin bootstrap: `src/Plugin.php` with a hand-rolled lazy service container (`register`/`has`/`get`) and a single `Plugin::boot()` entry point. `logscope.php` now defines `LOGSCOPE_PLUGIN_FILE`, requires Composer autoload, and hooks `Plugin::boot` on `plugins_loaded` priority 5. Fires the `logscope/booted` action once the container is built so extensions can register services. Text domain is loaded on `init`.
-   Lifecycle hooks: `Activator` seeds default options (`logscope_log_path`, `logscope_tail_interval`, `logscope_db_version`) and grants the `logscope_manage` capability to administrators. `Deactivator` clears Logscope-owned cron events. `uninstall.php` deletes all `logscope_*` options, transient prefixes, and removes `logscope_manage` from every role â€” works whether the plugin is active at the time of deletion or not.
-   Unit test scaffolding: `brain/monkey` dev dependency, `phpunit.xml.dist`, `tests/php/bootstrap.php`, and a shared `Logscope\Tests\TestCase` base class wiring Brain Monkey setup/teardown.
-   `Logscope\Support\Capabilities` helper with `required()` and `has_manage_cap(?int $user_id = null)`. The `logscope/required_capability` filter lets site owners remap the required cap; non-string / empty filter returns fall back to the default so authorization cannot be silently disabled.
